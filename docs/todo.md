# 流式查询
MySQL 原生协议中没有名为「流式查询」的独立命令或专属协议类型，所谓流式查询是应用层对**避免一次性加载全量结果到客户端内存、逐行/分批读取数据**的统称。协议层面通过两种不同的结果集读取机制实现流式效果：
1. 文本协议下的**无缓冲结果集（Use Result 模式）**
2. 二进制预处理协议下的**服务器端游标（Server-side Cursor 模式）**

---

## 一、文本协议：无缓冲结果集（Use Result 模式）
这是最基础、最常用的流式实现，对应 C API 中的 `mysql_use_result()`，JDBC 中 `setFetchSize(Integer.MIN_VALUE)` 就是基于此模式实现。

### 协议原理
1. **包结构与普通查询完全一致**
   客户端发送 `COM_QUERY`（命令码 0x03）包，服务端返回标准结果集序列：
   结果集头（列数）→ 列定义包 → EOF 包 → 行数据包（多个）→ 最终 EOF 包
   协议层面没有额外的标志位区分是否流式。

2. **核心差异是客户端读取策略**
   - 默认 `mysql_store_result()` 模式：客户端一次性从 TCP 缓冲区读取所有行数据，全量缓存到本地内存后再返回给应用。
   - `mysql_use_result()` 模式：客户端不做全量缓存，每次应用调用 `fetch_row` 时才从 socket 读取一行数据，仅保留当前行在内存中。

3. **服务端通过 TCP 流控实现限速**
   服务端本身仍会尽可能快地将所有行写入 TCP 发送缓冲区；当客户端读取慢导致 TCP 接收窗口被占满时，服务端发送线程会自动阻塞等待，通过 TCP 原生流控机制间接实现"按需发送"的效果。

### 核心特点
- 无需特殊协议协商，普通文本查询即可启用
- 服务端无额外状态开销，性能损耗低
- 连接独占：读取完所有行之前，同一连接不能执行其他查询
- 只能顺序向前读取，不支持回滚、跳行

---

## 二、二进制协议：服务器端游标（Server-side Cursor 模式）
这是 MySQL 协议层面明确定义的流式机制，基于预处理语句（Prepared Statement）实现，客户端可以主动控制每次拉取的行数。

### 前提条件
客户端握手阶段需声明支持 `CLIENT_PS_MULTI_RESULTS` 能力标志，以支持多结果集处理。

### 协议交互流程
#### 1. 预处理 SQL
客户端发送 `COM_STMT_PREPARE`（命令码 0x16）包，携带 SQL 语句；服务端返回 `statement_id` 及参数、列元数据。

#### 2. 执行并打开游标
客户端发送 `COM_STMT_EXECUTE`（命令码 0x17）包，在 `flags` 字段中指定游标类型：
| 标志值 | 枚举常量 | 说明 |
|--------|----------|------|
| 0x00 | `CURSOR_TYPE_NO_CURSOR` | 默认模式，全量返回结果 |
| 0x01 | `CURSOR_TYPE_READ_ONLY` | 只读游标，用于流式查询 |

服务端收到后执行查询，仅返回列定义元数据，不主动发送行数据，同时在服务端保留游标上下文。

#### 3. 分批拉取数据
客户端发送 `COM_STMT_FETCH`（命令码 0x1C）包主动拉取数据，包结构如下：
| 字节数 | 字段 | 说明 |
|--------|------|------|
| 1 | command | 固定为 0x1C |
| 4 | statement_id | 对应预处理语句的 ID |
| 4 | num_rows | 本次期望拉取的行数 |

服务端返回指定数量的二进制行数据包，最后以 EOF 包结束当前批次。

#### 4. 循环与关闭
- 客户端重复发送 `COM_STMT_FETCH`，直到最终 EOF 包携带 `SERVER_STATUS_LAST_ROW_SENT` 标志，表示所有数据读取完毕。
- 读取完成后客户端发送 `COM_STMT_CLOSE` 释放服务端游标资源。

### 核心特点
- 协议层面真正支持分批拉取，客户端完全控制读取节奏
- 服务端需保留游标状态，有一定内存和资源开销
- 支持自定义批次大小，平衡内存占用和网络交互次数
- MySQL 服务端游标仅支持只读、向前滚动，不支持滚动回退和可更新游标

---

## 三、两种流式模式的核心对比
| 维度 | Use Result 无缓冲模式 | 服务器游标模式 |
|------|------------------------|----------------|
| 协议基础 | 文本协议，无特殊字段 | 二进制预处理协议，有明确游标命令 |
| 控制方式 | 客户端读取节奏 + TCP 流控被动控制 | 客户端主动通过 FETCH 命令控制 |
| 服务端开销 | 极低，无状态保留 | 较高，需维护游标上下文 |
| 批次粒度 | 逐行读取 | 可自定义每次拉取行数 |
| 网络交互 | 连续流，无额外 RTT | 每批次一次 FETCH 请求，增加 RTT |
| 适用场景 | 大数据量全量遍历、追求低开销 | 需要控制读取节奏、分批业务处理 |

---

## 四、主流驱动的流式实现对应
- **JDBC (MySQL Connector/J)**：
  - `statement.setFetchSize(Integer.MIN_VALUE)` → 对应 Use Result 无缓冲流式模式
  - 连接参数 `useCursorFetch=true` + `setFetchSize(n)` → 对应服务器游标模式
- **Python (PyMySQL/MySQLdb)**：`SSCursor` 游标类型 → 对应 Use Result 无缓冲模式
- **MySQL 命令行客户端**：默认使用无缓冲模式，天然支持流式读取
