# SQL 方言透明转换中间件 需求规格说明书

## 1. 项目目标
开发一个 **SQL 方言自动转换中间件**，使业务系统用 **一套 SQL** 就能透明地访问 **多种不同数据库**，无需修改应用代码。  
产品形态：**JDBC 驱动包（Wrapper Driver）**，应用只需更换 JDBC URL 和驱动类名，即可自动完成 SQL 方言转换。

## 2. 功能需求

### 2.1 核心能力
- **SQL 方言解析**：能解析源数据库方言的 SQL 语句（DQL、DML、部分 DDL），生成统一的抽象语法树（AST）。
- **方言转换规则**：根据源方言和目标方言，自动将 AST 改写为目标方言的等价形式，并生成合法 SQL 字符串。
- **JDBC 透明拦截**：在 `Statement.execute()`、`Statement.executeQuery()` 等方法中拦截原始 SQL，经转换后执行。
- **连接透传**：除 SQL 改写外，所有 JDBC 方法（事务、元数据、结果集等）直接委托给底层真实数据库驱动。

### 2.2 支持的数据库范围（见下文分组）
初期支持以下 4 大源方言 → 4 大目标方言的任意互转：
- **PostgreSQL方言组**（10 个库）
- **MySQL方言组**（3 个库）
- **Oracle方言组**（4 个库）
- **SQL Server方言组**（1 个库）

后续扩展：SQLite、DB2 等独立方言。

### 2.3 支持的 SQL 子集（第一期）
- **SELECT** 查询，含 JOIN、子查询、UNION、GROUP BY、HAVING、ORDER BY
- **INSERT / UPDATE / DELETE**
- **分页**：`LIMIT/OFFSET`、`OFFSET FETCH`、`ROWNUM` 等相互转换
- **常用内置函数**（≥30 个）：字符串、日期、数学、聚合函数
- **标识符引用**自动转换（反引号、双引号、方括号等）
- **数据类型映射**（基础类型）

暂不支持：存储过程、复杂 DDL、窗口函数、递归 CTE、专有特殊语法。

## 3. 数据库方言分组（兼容性映射表）
根据 SQL 语法兼容性，将 21 种数据库归为以下几个 **方言组**。同一组内的数据库共享相同的转换规则。

### 3.1 PostgreSQL 方言组（10 个库）
| 序号 | 数据库         | 备注                         |
|------|----------------|------------------------------|
| 1    | PostgreSQL     | 源头                         |
| 2    | HighGo（瀚高） | PG 内核，高度兼容            |
| 3    | KingbaseES     | PG 内核（原生模式）          |
| 4    | PolarDB PG     | 阿里云 PolarDB PG 版本       |
| 5    | Greenplum      | MPP，PG 8.2 兼容             |
| 6    | TimescaleDB    | PG 时序扩展                  |
| 7    | PostGIS        | PG 空间扩展                  |
| 8    | IvorySQL       | PG 衍生的 Oracle 兼容库      |
| 9    | QuestDB        | 实现 PG 线协议，语法兼容 PG  |
| 10   | KaiwuDB        | 兼容 PG 生态                 |

> **转换标识符**：`dialect_group = "postgresql"`

### 3.2 MySQL 方言组（3 个库）
| 序号 | 数据库         | 备注                           |
|------|----------------|--------------------------------|

> **转换标识符**：`dialect_group = "mysql"`

| 序号 | 数据库           | 备注                                     |
|------|------------------|------------------------------------------|

### 3.4 SQL Server 方言组（1 个库）
| 序号 | 数据库      | 备注 |
|------|-------------|------|

> **转换标识符**：`dialect_group = "sqlserver"`

### 3.5 独立方言（暂不实现转换）
- **SQLite**：自成一系，待二期适配。
- **DB2**：自成一系，待二期适配。
- **MongoDB**：非关系型，不使用 SQL，本项目不做映射。
- **Kdb+（kdb）**：使用 q 语言，本项目不做映射。

## 4. 技术方案

### 4.1 整体架构
采用 **JDBC Wrapper 驱动** 模式，结构如下：

```
业务应用
   │
   ▼
DriverManager.getConnection("jdbc:translator:源方言:目标方言://真实库地址")
   │
   ▼
TranslatorDriver (实现 java.sql.Driver)
   │
   ▼
TranslatorConnection (包装真实连接)
   │
   ▼
TranslatorStatement / PreparedStatement
   │  在 execute() 内部：
   │    1. 捕获原始 SQL
   │    2. 调用翻译引擎，得到目标 SQL
   │    3. 通过真实 Statement 执行目标 SQL
   │    4. 返回真实 ResultSet
   │
   ▼
真实 JDBC 驱动（如 PostgreSQL Driver）
```

### 4.2 翻译引擎
- **选型**：Apache Calcite 作为 AST 解析与方言生成内核，搭配自定义 AST 改写规则处理复杂差异。
- **备选轻量方案**：JSqlParser 用于 SQL 解析与 AST 修改，适合初期快速验证，但最终推荐 Calcite 以获得更广的 SQL 覆盖率。
- **处理流程**：
  1. 使用 `SqlParser` 解析原始 SQL → `SqlNode`
  2. 应用一组 `SqlRewriter`（Visitor），根据源→目标方言改写 AST（函数名、分页语法、类型转换等）
  3. 使用目标 `SqlDialect` 生成目标 SQL 字符串

### 4.3 关键模块设计
- **DialectRegistry**：维护方言组到真实数据库产品名称的映射。
- **RewriteRuleEngine**：可插拔的规则集，允许按方言对注册转换规则。
- **SqlDialectFactory**：根据源/目标方言标识符获取对应的 Calcite `SqlDialect` 实例和改写规则链。
- **JdbcUrlParser**：解析特殊 JDBC URL，提取“源方言”、“目标方言”、“真实连接参数”。

## 5. 转换规则示例（核心差异）
以下差异规则必须实现，单元测试需覆盖。

| 转换类型       | 源方言示例                  | 目标方言示例                       |
|----------------|-----------------------------|------------------------------------|
| 分页           | `LIMIT 10 OFFSET 5` (MySQL) | `OFFSET 5 ROWS FETCH NEXT 10 ROWS ONLY` (Oracle) |
| 分页           | `ROWNUM <= 20` (Oracle)     | `LIMIT 10 OFFSET 10` (PG)          |
| 字符串拼接     | `CONCAT(a,b)` (MySQL)       | `a \|\| b` (PG/Oracle)             |
| 空值处理       | `IFNULL(a,0)` (MySQL)       | `COALESCE(a,0)` (PG)               |
| 空值处理       | `NVL(a,0)` (Oracle)         | `COALESCE(a,0)` (PG/MySQL)         |
| 标识符引用     | `` `table` `` (MySQL)       | `"table"` (PG)                     |
| 标识符引用     | `[table]` (SQL Server)      | `"table"` (PG)                     |
| 日期时间函数   | `NOW()` (MySQL)             | `CURRENT_TIMESTAMP` (PG)           |
| 日期时间函数   | `GETDATE()` (SQL Server)    | `NOW()` (MySQL) / `CURRENT_TIMESTAMP` (PG) |

## 6. 非功能性需求
- **性能**：SQL 转换耗时 < 5ms（典型查询），对整体查询延迟影响可忽略。
- **可扩展性**：新增方言只需增加一组改写规则和方言定义，无需修改核心代码。
- **测试要求**：每种方言组合需基于 Testcontainers 启动真实数据库进行集成测试。
- **文档**：需提供支持的 SQL 函数列表、不支持的边界说明。

## 7. 开发阶段建议

### Phase 1：翻译核心（1～2 周）
- 基于 Calcite 搭建解析→改写→生成流程
- 实现 PG ↔ MySQL 之间的基础 SELECT 转换（含分页、常用函数）
- 编写单元测试，使用内存数据库或固定 SQL 对比

### Phase 2：JDBC Wrapper 驱动（1 周）
- 实现 TranslatorDriver / Connection / Statement
- 集成翻译核心
- 集成测试（Spring Boot 应用同时连接 MySQL 和 PG，用同一套 SQL 查询）

### Phase 3：扩展方言与规则（2 周）
- 逐步加入 Oracle、SQL Server 方言及互转规则
- 处理 PreparedStatement 参数索引适配（如需支持）
- 适配 DML 语句（INSERT/UPDATE/DELETE）

### Phase 4：组内兼容性验证（1 周）
- 对分组内的其他数据库（HighGo、Kingbase、Doris 等）进行实测，修正微小差异
- 形成最终方言-映射配置表
